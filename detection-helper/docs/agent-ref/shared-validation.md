# Cross-Phase Validation Rules

> **Loaded by:** `detection-helper.agent.md` (Path C) before finalizing any rule.
> **Purpose:** Reconcile the authored rule against what earlier phases (Path A
> decomposition, Path B telemetry) confirmed. Prevents silent disagreements
> (e.g., telemetry says `FileName`, rule uses `ProcessName`).

## Mandatory validation gate

Before finalizing ANY rule, Path C must:

### Step 1: Read session state

```markdown
- [ ] Read `docs/session/state.md`
- [ ] If Phase 2 (telemetry verification) exists, compare every table/ActionType/column
- [ ] If no session state, run own telemetry verification (embedded workflow)
```

### Step 2: Cross-check every table reference

```markdown
- [ ] Table in rule == table in session state (or confirmed via index)
- [ ] Mismatch found → STOP, flag discrepancy, do not write rule
```

### Step 3: Cross-check every ActionType

```markdown
- [ ] ActionType string matches exactly (case-sensitive) session state or `telemetry-index.json`
- [ ] ActionType appears in `docs/tenant/all-actiontypes.md` or flagged as "documented only"
- [ ] Mismatch found → STOP, flag discrepancy
```

### Step 4: Cross-check every column

```markdown
- [ ] Column appears in `docs/advanced-hunting/<table>-table.md`
- [ ] Column NOT in `docs/notes/anti-patterns.md` as known trap
- [ ] Generic names have correct prefix (`InitiatingProcess*` not bare name)
- [ ] Column confidence ≥ L3 (schema-confirmed)
- [ ] L1/L2 column → mark as "proposed — verify before deploying"
```

### Step 5: Cross-check primitives

```markdown
- [ ] Primitive in rule == primitive from threat-translator (if pipeline used)
- [ ] Kill-chain position matches priority order in session state
- [ ] Gap acknowledged → rule includes fallback or caveat
```

### Step 6: Confidence reconciliation

```markdown
- [ ] Rule confidence ≤ min(telemetry confidence, pattern confidence)
- [ ] Example: telemetry says L3 (schema-confirmed), rule cannot claim L4
- [ ] If rule requires L4 but telemetry is only L3 → include validation query
```

## Discrepancy handling

When a mismatch is detected:

```markdown
**DISCREPANCY DETECTED — Rule halted.**

| Element | Telemetry says | Rule uses | Status |
|---|---|---|---|
| Column | `FileName` [L3] | `ProcessName` [L1] | MISMATCH |
| ActionType | `ProcessCreated` [L4] | `ProcessCreated` [L4] | OK |

Resolution required:
1. Use `FileName` instead of `ProcessName`
2. Verify `FileName` is the correct column for this detection
3. Re-run validation gate after fix
```

## Session state write-back

After passing validation, append to `docs/session/state.md` Phase 3:

```yaml
rules:
  - rule_id: "<auto-generated>"
    name: "<rule name>"
    primitive: "<P1-P12>"
    pattern_source: "<pattern file or 'custom'>"
    languages: [kql, spl]
    confidence_level: "<L1-L5>"
    # L-level = minimum of all cross-checked elements
    quality_score:
      coverage: "high|medium|low"
      fp_risk: "low|medium|high"
      performance: "fast|moderate|slow"
    validated_against_session: true
    discrepancies: []  # empty if clean
```
